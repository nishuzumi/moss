# Every ABI has a documented origin: compiled > explorer > vendored

ABIs are load-bearing security artifacts — one wrong function signature means wrong calldata means wrong fund flows. Every ABI in the repo lives in the package's `src/abis/` folder with an **ABI origin** header naming exactly one provenance tier:

1. **compiled** — generated from contract source in the package (foundry project in `contracts/`, `pnpm gen:abis` runs `forge build` and emits the TS with an auto-stamped header: source file, solc version, date). The strongest tier; used by `@themoss/erc` for the ERC standard interfaces.
2. **explorer** — pulled from a block explorer's *verified* contract page; header records the URL and retrieval date. The root `pnpm fetch-abi` command uses the official Etherscan V2 endpoint for Monad mainnet (`chainid=143`, `action=getabi`) and emits the full ABI as typed `as const` TypeScript; see [Protocol onboarding](../protocol-onboarding.md#fetch-an-explorer-abi).
3. **vendored** — taken from a third-party verifiable source (official SDK on npm, protocol repo). Hand-copying is NOT vendoring: the package commits upstream files **verbatim** in `abis-src/` and deterministically regenerates the full TS artifact, so reviewers diff against upstream instead of trusting a transcription. Follow `dist-tags.latest` with a 7-day release-age guard, never highest-semver; the header records package@version, tarball sha256, and on-chain verification.

If none of the three is possible, the protocol does not get an adapter.

## Mechanics worth recording

- **Generated ABIs are committed TypeScript, not JSON.** abitype's type inference — the thing that makes Handles fully typed — requires `as const` literals, which JSON imports cannot provide. Generation scripts emit `export const XAbi = [...] as const` files; human-readable `parseAbi` string arrays are equally acceptable for explorer/vendored tiers.
- Contributors never need foundry: compiled ABIs are regenerated only when `contracts/` changes. Foundry projects (`contracts/`, `out/`, `cache/`) are excluded from npm (`files: ["dist"]`) and build outputs are gitignored.
- Tooling per tier: the compiled tier uses the ecosystem-standard **@wagmi/cli foundry plugin** (`gen:abis` = `wagmi generate`; the committed `wagmi.config.ts` + `contracts/` are the provenance record). Gotcha worth knowing: wagmi's default excludes filter out `IERC20.sol/**` and other common interface names — override `exclude` when the interface is your own source of truth. The vendored tier keeps a thin in-repo script (`update:abis`) because no standard tool covers npm-tarball vendoring with digest recording.
- **Generated ABIs are FULL upstream artifacts.** An earlier revision emitted per-function allowlists as a least-privilege measure; superseded — the friction outweighed the defense-in-depth, given where enforcement actually lives: capabilities are reviewed where they are written, while exhaustive Receipt parsing and Agent intent alignment expose simulated behavior before signing. Trim an ABI only if tsc type-instantiation cost on a giant artifact demonstrably hurts.
- **The source→generated link is test-enforced, not asserted.** Vendored packages commit machine-readable provenance (`abis-src/VENDOR.json`: package, version, tarball sha256, vendored date, release-age guard) and derive the TS deterministically from committed inputs (`gen:abis`, offline — no clock, no network). A unit test asserts the committed TS equals the derivation byte-for-byte, so hand-edits to generated files, or source/generated drift, fail the suite. Generated `src/abis/**` are excluded from the formatter for the same reason.
- **Foundry-in-monorepo policy: git submodules are banned; `forge init` and `forge install` MUST carry `--no-git`.** Foundry's git coupling lives in the submodule model (registered at the *git root* — repo-wide pollution, `submodule update --init` clone friction, a second dependency system beside pnpm); `forge build` itself needs no git and works nested (verified in this repo). For external Solidity dependencies the preferred sources are, in order: (1) npm package + `remappings.txt` into node_modules — versioned, single package manager; (2) `forge soldeer` — foundry's native non-git manager; (3) `forge install --no-git` — plain-file vendoring into `lib/` (unversioned). The invariant is enforced in CI: the build fails if `.gitmodules` or any gitlink entry exists.

## Consequences

- The origin header is part of the adapter Definition of Done and reviewed like code.
- "The ABI came from my memory of the standard" is not an origin. Even ERC-20 goes through solc.
