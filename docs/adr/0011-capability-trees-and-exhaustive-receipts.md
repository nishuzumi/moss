# Capability trees and exhaustive Receipts are the write verification model

A write `action` returns one Capability tree; each Capability owns exactly one immediate unsigned transaction and one typed Receipt parser, while extra transactions belong to nested Capabilities and core alone validates and depth-first flattens the tree. Simulation gives each successful transaction's pure parser every raw Event and native transfer in provable order, then verifies that the recursive Receipt retains the exact original Change objects with complete identity, length, and order; a revert yields no Receipt, preserves earlier Receipts, and stops later transactions. A Receipt translates only facts directly supported by its Changes: it never reconstructs the planned path or consults external state to fill evidence gaps. The library Simulator returns complete evidence for SDK consumers; the MCP adapter exposes only verified leaf texts in exact Change order plus Warnings for Agent intent alignment. Declarations alone are not evidence.
