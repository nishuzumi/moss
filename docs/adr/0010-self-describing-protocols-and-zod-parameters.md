# Protocol packages are self-describing and compose through injected dependencies

A Protocol package exports top-level self-describing `@Protocol` classes; the composition root supplies selected modules, and Registry registers their Protocol exports and recursively injects declared dependencies. Core owns framework contracts, simulator owns trace mechanics, Protocol packages own ABI semantics, Receipts, and protocol-exclusive deployments, `@themoss/erc` stays address-free, system owns the shared Monad Runtime and shared verified asset constants and rejects non-143 RPCs, and MCP server owns transport; dynamic addresses come from chain state and adding a Protocol changes only its package and composition root. Capability, Query, and optional Protocol-binding inputs use `{ type, description }`, separating a reusable context-free Zod value contract from the field's purpose and exposing the generated JSON-safe schemas through `load`.

Protocol classes compose through declared `protocols` dependencies and typed injected fields. An ordinary dependency injects a typed Protocol reference. A parameterized dependency exports a `protocolFactory` alias and injects an uncached factory: `create(binding)` synchronously validates the binding before Protocol execution or RPC, derives that instance's Handles, and returns only bound Capabilities and Queries; `receipts` returns only pure parsers and needs no binding. Binding is distinct from method parameters and is retained canonically in each produced CapabilityNode. A decorated Protocol cannot extend another decorated Protocol; Registry rejects that inheritance so Receipt-only instances never execute a parent Protocol's Runtime-injecting constructor.
