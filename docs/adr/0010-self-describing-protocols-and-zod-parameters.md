# Protocol packages are self-describing and compose through injected dependencies

A Protocol package exports top-level self-describing `@Protocol` classes; the composition root supplies selected modules, and Registry registers their Protocol exports and recursively injects declared dependencies. Core owns framework contracts, simulator owns trace mechanics, Protocol packages own ABI semantics, Receipts, and protocol-exclusive deployments, `@themoss/erc` stays address-free, system owns the shared Monad Runtime and shared verified asset constants and rejects non-143 RPCs, and MCP server owns transport; dynamic addresses come from chain state and adding a Protocol changes only its package and composition root. Capability and Query inputs use `{ type, description }`, separating a reusable context-free Zod value contract from the field's purpose and exposing the generated JSON-safe schema through `load`.

Protocol classes compose through declared `protocols` dependencies and typed injected fields. A decorated Protocol cannot extend another decorated Protocol; Registry rejects that inheritance so Receipt-only instances never execute a parent Protocol's Runtime-injecting constructor.

Protocol metadata may declare fixed Package address labels independently of Handles. Registry prefixes each local label with the title-cased Protocol slug and validates the final 1–32 character safe name. The composition root supplies Trusted token labels through the explicit Registry constructor option; Registry never discovers them by scanning ordinary Protocol exports. A root Protocol sees only its own Package labels and those of direct or transitive declared dependencies.
